# SOLID Findings - FootballManager BugHunt

| SOLID Principle | Finding in BugHunt Code | Why It Violates SOLID | Refactoring Idea | Refactored Code Element |
|---|---|---|---|---|
| **S - Single Responsibility Principle** | `FootballManager` selects strategies, trains players, saves players and sends notifications. | The class has multiple reasons to change: strategy logic, training logic, persistence logic and notification logic. | Keep `FootballManager` as coordinator only and move details into dedicated classes. | `TrainingPlan`, `MatchStrategy`, `PlayerRepository`, `PlayerNotifier` |
| **O - Open/Closed Principle** | `FootballManager::select_strategy()` uses `if/else` checks for strategy strings. | Adding a new strategy requires modifying existing `FootballManager` code. | Replace string-based decisions with polymorphic strategy classes. | `MatchStrategy`, `OffensiveStrategy`, `DefensiveStrategy` |
| **L - Liskov Substitution Principle** | `InjuredPlayer::train(int intensity)` accepts only values up to `30`, while `Player::train(int intensity)` accepts values from `0` to `100`. | The derived class strengthens the precondition and can no longer safely replace the base class. | Remove the problematic inheritance and model injury as state or use separate training plans. | `Player` with `is_injured()`, `NormalTrainingPlan`, `RecoveryTrainingPlan` |
| **I - Interface Segregation Principle** | `ClubService` forces one interface to contain training, saving and notification methods. | A class that only wants one responsibility would still depend on unrelated methods. | Split the large interface into smaller role-specific abstractions. | `TrainingPlan`, `PlayerRepository`, `PlayerNotifier` |
| **D - Dependency Inversion Principle** | `FootballManager` directly owns `FilePlayerRepository` and `EmailNotifier`. | High-level coordination logic depends on concrete low-level details. | Depend on abstractions and inject concrete implementations from outside. | Constructor injection with `PlayerRepository&` and `PlayerNotifier&` |
